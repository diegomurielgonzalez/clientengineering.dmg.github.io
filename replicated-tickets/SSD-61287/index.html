Investigation & Replication Steps

Below is the summary and solution provided to the client for setting up the necessary Business Rules to resolve this UI glitch.

The Root Cause

The client website functions as a Single Page Application (SPA). This means it dynamically updates content without ever doing a hard page reload. When visitors navigate away from the /subdomain1/auth page, the website instantly destroys the authentication window.

Because the window's "Close" button vanishes before anyone actually clicks it, the original rule designed to reset the bubble's position never fires. The bubble stays up simply because it never received the signal to come back down.

The Solution

To fix this, a failsafe Web Business Rule is required. Instead of relying on a button click, Glia is configured to automatically force the bubble back to its default position anytime the user leaves the authentication page.

Step-by-Step Configuration

Step 1: Set the Rule Triggers
  • Go to Admin Console > Site Settings > Business Rules > Web and click Add new.
  • Enter a clear name for the rule (e.g., "Failsafe: Reset Bubble Position").
  • Under Pre-conditions, select Coming From.
  • Enter (!*/subdomain1/auth*)* in the input field. This specific formatting tells Glia to exclude the auth page and trigger the rule the moment the user leaves it.
Step 2: Catch the SPA Navigation
  • Scroll down to Source and select Navigation.
  • Enter * in the input field. This acts as a wildcard, allowing Glia to detect whenever the SPA silently changes the URL to a new path.
Step 3: Force the Position Reset
  • Under Actions, select Set reactive bubble position.
  • Explicitly define the default pixel values to override the elevated state. Enter 15px for the vertical position and 20px for the horizontal position.
  • Click Save.

This failsafe makes the layout bulletproof. Regardless of how fast visitors navigate away from the authentication page, the bubble will instantly snap back to its proper place.
