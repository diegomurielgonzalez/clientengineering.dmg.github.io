Investigation Summary

The client website functions as a Single Page Application (SPA). This means it dynamically updates content without ever doing a hard page reload. When visitors navigate away from the /subdomain1/auth page, the website instantly destroys the authentication window.

Because the window's "Close" button vanishes before anyone actually clicks it, the original rule designed to reset the bubble's position never fires. The bubble stays up simply because it never received the signal to come back down.

Solution & Web Business Rules Setup

To fix this, three interconnected Web Business Rules are required. Rules 1 and 2 handle the standard behavior (opening and closing the modal), while Rule 3 acts as the failsafe to automatically force the bubble back to its default position anytime the user leaves the authentication page.

Rule 1: Move bubble up when div appears

  • Conditions > Coming From: *
  • Source: Clicking as id auth-btn
  • Actions > Set reactive bubble position: 320px vertically, 20px horizontally

Rule 2: Move bubble down when div disappears

  • Conditions > Coming From: *
  • Source: Clicking as id close-modal-btn
  • Actions > Set reactive bubble position: 15px vertically, 20px horizontally

Rule 3: Return bubble to original position

  • Conditions > Coming From: (!*/subdomain1/auth*)* (This formatting tells Glia to exclude the auth page and trigger the rule the moment the user navigates away from it)
  • Source: Navigation * (This acts as a wildcard, allowing Glia to detect whenever the SPA silently changes the URL)
  • Actions > Set reactive bubble position: 15px vertically, 20px horizontally

This failsafe makes the layout bulletproof. Regardless of how fast visitors navigate away from the authentication page, the bubble will instantly snap back to its proper place.
